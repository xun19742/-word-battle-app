# WordRush 学习打卡记录设计

## 目标

在现有背词 MVP 上增加一个轻量的学习打卡记录，让用户完成任意一轮学习后，可以在首页看到当天是否已打卡，并在独立页面查看最近学习日期、学习词数、得分和连续打卡天数。

本期只做本地 MVP，不做云端同步、不做朋友圈海报、不做复杂日历视图。优先保证背词流程不被打断，记录稳定可测，后续再把数据接入云端和排行榜。

## 用户流程

1. 用户从首页开始学习新词或复习词。
2. 完成一轮后进入总结页。
3. 总结页写入学习记录，同时自动更新当天打卡。
4. 用户返回首页，首页显示：
   - 今日已打卡或今日未打卡。
   - 连续打卡天数。
   - 今日学习词数。
5. 用户点击首页“打卡记录”，进入记录页。
6. 记录页按日期倒序展示每天学习情况。

打卡由完成学习自动触发，不额外要求用户点“打卡”按钮。这样更符合背词 App 的直觉：学了就算打卡。

## 页面结构

新增页面：

- `pages/checkin/index`：展示打卡记录列表和基础统计。

首页新增区域：

- 打卡卡片：展示今日状态、连续天数、今日词数。
- “查看打卡记录”入口。

总结页不新增复杂 UI，只在完成记录后展示一句轻提示，例如“今日打卡已记录”。如果后续测试发现总结页太拥挤，可以只保留首页入口。

## 数据模型

新增本地存储 key：

```js
wordrush.checkins
```

存储结构：

```js
{
  days: {
    "2026-06-25": {
      date: "2026-06-25",
      completed: 20,
      newCompleted: 10,
      reviewCompleted: 10,
      score: 160,
      rounds: 2,
      lastCheckedAt: "2026-06-25T12:30:00.000Z"
    }
  }
}
```

字段含义：

- `completed`：当天学习总词数。
- `newCompleted`：当天新词学习数量。
- `reviewCompleted`：当天复习数量。
- `score`：当天累计得分。
- `rounds`：当天完成轮次数。
- `lastCheckedAt`：当天最后一次完成学习的时间。

旧数据或异常数据按空记录处理，避免页面崩溃。

## 客户端服务

新增 `miniprogram/services/checkin-service.js`，提供：

- `applyCheckinSummary(summary)`：根据一轮学习总结更新当天打卡。
- `loadCheckinStats(today)`：返回首页展示所需数据。
- `listCheckinDays(limit)`：返回记录页列表，默认最近 30 天。
- `normalizeCheckinState(value)`：清洗本地缓存，方便测试。

服务层只依赖 `wx.getStorageSync`、`wx.setStorageSync` 和传入的时间函数，不直接依赖页面。

## 打卡规则

- 完成一轮学习才生成打卡。
- 同一个 `roundId` 不重复打卡，避免总结页重复进入后累计两次。
- 新词和复习都算打卡。
- 当天多轮学习会累加词数、得分和轮次数。
- 连续打卡天数按自然日倒推计算，只要某天 `completed > 0` 就算打卡。
- 今日未学习时，首页显示“今日未打卡”，连续天数按最近连续记录计算。

## 接入点

总结页 `pages/summary/index.js` 当前已经在 `onLoad()` 中：

1. 读取已完成轮次。
2. 生成 summary。
3. 调用 `createWxLearningRepository().applySummary(summary)`。
4. 清理当前轮次。

打卡服务接在第 3 步之后：

```js
applyCheckinSummary(summary)
```

如果学习记录因为重复 `roundId` 没有应用，打卡也不应重复应用。实现时可以在打卡服务内部维护 `processedRoundIds`，保证幂等。

## 错误处理

- 本地缓存为空：返回空打卡状态。
- 本地缓存结构损坏：自动回落为空状态。
- `summary` 缺少 `roundId` 或 `total`：忽略并返回 `{ applied: false }`。
- 页面列表为空：显示“完成一轮学习后，这里会出现你的打卡记录”。

## 测试策略

继续使用现有 `node:test`：

- 服务测试：首次打卡、同一天多轮累加、重复 `roundId` 不重复累计。
- 连续天数测试：连续日期正确计数，中断日期停止计数。
- 首页结构测试：存在打卡卡片、打卡入口和 `openCheckins()`。
- 页面结构测试：新增页面注册、记录页展示空状态和列表字段。
- 总结页测试：完成学习后调用打卡服务。

## 实施顺序

1. 新增打卡服务和服务测试。
2. 总结页接入自动打卡。
3. 首页展示打卡状态和入口。
4. 新增打卡记录页。
5. 更新 README 和发布前验收清单。

这个顺序保证底层规则先可测，再接 UI，最后补文档。
